## 1. Ant Design 组件测试

### 1.1 问题：Ant Design 组件的特殊 DOM 结构

Ant Design 组件经常有复杂的 DOM 结构和特殊的可见性控制。

#### ❌ 错误示例

```typescript
// 直接检查容器的可见性
await expect(page.locator('.ant-table-tbody')).toBeVisible();
```

**失败原因：**
- Ant Design Table 使用虚拟滚动，tbody 可能有 `visibility: hidden` 样式
- 元素在 DOM 中但不一定可见

#### ✅ 正确示例

```typescript
// 检查实际内容元素的附加状态
await expect(page.locator('.ant-table-tbody tr[data-row-key]').first()).toBeAttached();

// 或检查具体的列头
await expect(page.getByRole('columnheader', { name: '题型' })).toBeAttached();
```

**关键原则：**
- 优先使用 `toBeAttached()` 而不是 `toBeVisible()` 检查元素存在
- 检查具体内容而不是容器
- 使用 semantic locators（`getByRole`, `getByLabel` 等）

## 2. 表格测试

### 2.1 问题：虚拟滚动表格中的元素不可见

Ant Design Table 默认启用虚拟滚动，导致不在视口内的行不可见。

#### ❌ 错误示例

```typescript
// 尝试直接操作表格行中的按钮
const editButton = page.locator('button:has-text("编辑")').first();
await editButton.click(); // 可能失败：元素不可见
```

**失败原因：**
- 虚拟滚动只渲染可见区域的行
- 按钮可能在 DOM 中但在视口外

#### ⚠️ 部分有效的方案

```typescript
// 方案1: 先滚动到行，再操作 - 对虚拟滚动表格可能失败！
const firstRow = page.locator('.ant-table-tbody tr[data-row-key]').first();
await firstRow.scrollIntoViewIfNeeded(); // ⚠️ 虚拟滚动表格中此方法可能超时
await expect(firstRow).toBeAttached();

const editButton = firstRow.getByRole('button', { name: /编辑/ }).first();
await editButton.waitFor({ state: 'visible', timeout: 5000 });
await editButton.click();

// 方案2: 使用 data-row-key 精确定位 - 仍然受虚拟滚动限制
const row = page.locator('tr[data-row-key="123"]');
await row.scrollIntoViewIfNeeded(); // ⚠️ 可能失败
```

#### ✅ 最佳方案：使用 JavaScript evaluate() 绕过可见性检查

```typescript
// 等待表格行加载（确保DOM中有数据）
const tableRows = page.locator('.ant-table-tbody tr[data-row-key]');
await expect(tableRows.first()).toBeAttached({ timeout: 5000 });

// 定位目标按钮（使用 aria-label 或其他稳定属性）
const editButtons = page.locator('button:has([aria-label="edit"])');
await editButtons.first().waitFor({ state: 'attached', timeout: 5000 });

// 使用 evaluate() 在浏览器上下文中直接执行点击
// 这会绕过 Playwright 的可见性检查
await editButtons.first().evaluate((button: HTMLElement) => button.click());
```

**为什么这种方法有效：**
1. **绕过可见性限制** - `evaluate()` 直接在浏览器DOM中执行，不受Playwright可见性规则限制
2. **适用于虚拟滚动** - 只要元素在DOM中（attached），就可以点击
3. **可靠性高** - 不依赖 `scrollIntoViewIfNeeded()` 的虚拟滚动支持

**何时使用：**
- Ant Design Table 等启用虚拟滚动的组件
- 元素在DOM中但不在视口内
- `scrollIntoViewIfNeeded()` 或 `force: true` 失败时

**何时不使用：**
- 标准可见元素（优先使用 `.click()`）
- 需要真实用户交互的场景（如hover触发的菜单）
- 元素完全不在DOM中的情况

**关键原则：**
- 始终先检查元素是否 attached
- 对于需要交互的元素，使用 `scrollIntoViewIfNeeded()`
- 从父容器（行）开始定位子元素（按钮）
- 添加适当的等待和超时

### 2.2 表格列验证

#### ❌ 错误示例

```typescript
await expect(page.locator('th:has-text("题型")')).toBeVisible();
```

**问题：**
- 文本匹配不精确
- 可能匹配到多个元素

#### ✅ 正确示例

```typescript
// 使用 role-based selector
await expect(page.getByRole('columnheader', { name: '题型' })).toBeAttached();
await expect(page.getByRole('columnheader', { name: '科目' })).toBeAttached();
await expect(page.getByRole('columnheader', { name: '年级' })).toBeAttached();
```

**关键原则：**
- 使用语义化选择器 `getByRole('columnheader')`
- 精确匹配列名
- 使用 `toBeAttached()` 检查存在性

## 3. Checkbox/Radio 测试

### 3.1 问题：Checkbox 状态检查失败

Ant Design 的 Checkbox 在已选中状态下尝试 `check()` 会失败。

#### ❌ 错误示例

```typescript
// 直接 check，可能已经选中
await page.check('label:has-text("练习题库") input[type="checkbox"]');
// ❌ Error: Clicking the checkbox did not change its state

// 使用 setChecked 也可能失败
await checkbox.setChecked(true);
// ❌ Error: Clicking the checkbox did not change its state
```

**失败原因：**
- Playwright 的 `check()` 和 `setChecked()` 要求状态必须改变
- Ant Design Checkbox 可能因为动画或内部状态导致 Playwright 认为状态未变

#### ✅ 正确示例

```typescript
// 方案1: 使用 evaluate 检查原生状态后点击 label
const checkbox = page.locator('label:has-text("练习题库") input[type="checkbox"]');
const isChecked = await checkbox.evaluate((el: HTMLInputElement) => el.checked);

if (!isChecked) {
  // 点击 label 而不是 checkbox
  await page.locator('label:has-text("练习题库")').click();
  await page.waitForTimeout(300); // 等待动画完成
}

// 方案2: 使用 force 点击（不推荐，除非必要）
await checkbox.click({ force: true });

// 方案3: 直接点击 label (最简单)
await page.locator('label:has-text("练习题库")').click();
```

**关键原则：**
- 使用 `evaluate()` 检查原生 DOM 状态，而不是 `isChecked()`
- 点击 label 而不是 input
- 给 Ant Design 动画留出时间（200-300ms）
- 避免依赖 `check()` 和 `setChecked()` 用于 Ant Design 组件

### 3.2 多选场景

#### ✅ 正确示例

```typescript
// 处理多个 checkbox
const checkboxes = [
  { selector: 'label:has-text("练习题库")', input: 'label:has-text("练习题库") input[type="checkbox"]' },
  { selector: 'label:has-text("测评题库")', input: 'label:has-text("测评题库") input[type="checkbox"]' }
];

for (const cb of checkboxes) {
  const input = page.locator(cb.input);
  const isChecked = await input.evaluate((el: HTMLInputElement) => el.checked);

  if (!isChecked) {
    await page.locator(cb.selector).click();
    await page.waitForTimeout(300);
  }
}
```

## 4. 选择器策略

### 4.1 选择器优先级

**推荐顺序（从高到低）：**

1. **Role-based selectors（最优）**
   ```typescript
   page.getByRole('button', { name: '提交' })
   page.getByRole('columnheader', { name: '题型' })
   page.getByRole('textbox', { name: '用户名' })
   ```

2. **Label/Placeholder（次优）**
   ```typescript
   page.getByLabel('用户名')
   page.getByPlaceholder('请输入用户名')
   ```

3. **Test ID（推荐用于动态内容）**
   ```typescript
   page.getByTestId('submit-button')
   ```

4. **CSS Selectors（仅当以上都不适用）**
   ```typescript
   page.locator('.ant-table-tbody tr[data-row-key]')
   ```

5. **Text-based（最不推荐，易变）**
   ```typescript
   page.locator('button:has-text("提交")') // 避免使用
   ```

### 4.2 组合选择器

#### ✅ 正确示例

```typescript
// 从父容器开始，逐层定位
const row = page.locator('.ant-table-tbody tr[data-row-key]').first();
const editButton = row.getByRole('button', { name: /编辑/ });

// 使用 filter 精确匹配
const practiceCheckbox = page.locator('label')
  .filter({ hasText: '练习题库' })
  .filter({ has: page.locator('input[type="checkbox"]') });

// 提供备选方案
const button = page.getByRole('button', { name: '编辑' })
  .or(page.getByLabel('edit'))
  .first();
```

## 5. 等待策略

### 5.1 等待时机

#### ❌ 错误示例

```typescript
// 直接操作，不等待
await page.click('.ant-tabs-tab:has-text("我的草稿")');
const button = page.locator('button:has-text("编辑")');
await button.click(); // 可能失败：元素还未加载
```

#### ✅ 正确示例

```typescript
// 等待页面稳定
await page.goto('/teacher/question-bank');
await page.waitForLoadState('networkidle');

// 切换 tab
await page.click('.ant-tabs-tab:has-text("我的草稿")');
await page.waitForTimeout(1000); // 等待 tab 切换完成

// 等待内容加载
const tableRows = page.locator('.ant-table-tbody tr[data-row-key]');
await expect(tableRows.first()).toBeAttached({ timeout: 5000 });

// 等待元素可见后操作
const button = tableRows.first().getByRole('button', { name: /编辑/ });
await button.waitFor({ state: 'visible', timeout: 5000 });
await button.click();
```

### 5.2 超时设置建议

```typescript
// 定义超时常量
export const TEST_TIMEOUTS = {
  PAGE_LOAD: 30000,      // 页面加载: 30秒
  NETWORK_IDLE: 10000,   // 网络空闲: 10秒
  ELEMENT_WAIT: 10000,   // 元素等待: 10秒
  ANIMATION: 500,        // 动画完成: 500ms
  MODAL: 3000,          // 模态框出现: 3秒
  MESSAGE: 5000,        // 消息提示: 5秒
  API_RESPONSE: 15000   // API响应: 15秒
};

// 使用示例
await expect(page.locator('.ant-message-success')).toBeVisible({
  timeout: TEST_TIMEOUTS.MESSAGE
});
```

### 5.3 等待网络请求完成

```typescript
// 等待特定 API 请求
await page.waitForResponse(
  response => response.url().includes('/api/question-bank') && response.status() === 200,
  { timeout: TEST_TIMEOUTS.API_RESPONSE }
);

// 或等待网络空闲
await page.waitForLoadState('networkidle');
```

## 6. 模态框处理

### 6.1 问题：模态框阻挡其他元素

#### ❌ 错误示例

```typescript
// 提交后立即点击其他元素
await submitButton.click();
await page.click('.ant-tabs-tab:has-text("待我审核")'); // 失败：模态框还在
```

**失败原因：**
- 模态框还未关闭，阻挡了背后的元素
- 成功消息提示在显示，阻挡了点击

#### ✅ 正确示例

```typescript
// 方案1: 等待模态框消失
await submitButton.click();
await expect(page.locator('.ant-modal')).not.toBeVisible({ timeout: 3000 });
await page.click('.ant-tabs-tab:has-text("待我审核")');

// 方案2: 显式关闭模态框
await submitButton.click();
await page.waitForTimeout(1000);

const modalClose = page.locator('.ant-modal-close');
if (await modalClose.isVisible({ timeout: 1000 }).catch(() => false)) {
  await modalClose.click();
  await page.waitForTimeout(500);
}

// 方案3: 等待成功消息后操作
await submitButton.click();
await expect(page.locator('.ant-message-success')).toBeVisible({ timeout: 5000 });
await expect(page.locator('.ant-message-success')).not.toBeVisible({ timeout: 5000 });
await page.click('.ant-tabs-tab:has-text("待我审核")');
```

### 6.2 问题：模态框提交后不关闭

有时模态框提交后不会自动关闭，可能是因为表单验证失败或其他原因。

#### ❌ 错误示例

```typescript
// 提交后直接等待模态框消失
await submitButton.click();
await expect(page.locator('.ant-modal')).not.toBeVisible({ timeout: 5000 }); // 可能超时失败
```

**失败原因：**
- 表单验证失败，模态框显示错误信息
- 提交操作触发了其他交互（如确认对话框）
- 网络请求失败，模态框保持打开状态

#### ✅ 正确示例

```typescript
// 提交并处理可能的错误
const submitButton = page.locator('.ant-modal button:has-text("提交")');
await submitButton.click();

// 等待一段时间让请求完成
await page.waitForTimeout(2000);

// 检查是否有表单验证错误
const errorMsg = page.locator('.ant-modal .ant-form-item-explain-error');
if (await errorMsg.isVisible().catch(() => false)) {
  // 如果有错误，手动关闭模态框
  const closeBtn = page.locator('.ant-modal-close');
  await closeBtn.click();
}

// 使用较长的超时等待模态框关闭
await expect(page.locator('.ant-modal')).not.toBeVisible({ timeout: 10000 });
```

**关键点：**
- 先等待足够时间让操作完成
- 检查是否有错误消息导致模态框未关闭
- 必要时手动关闭模态框
- 使用较长的超时时间（10秒而不是3-5秒）

### 6.3 模态框内操作

```typescript
// 等待模态框出现
await expect(page.locator('.ant-modal')).toBeVisible({ timeout: 3000 });
await expect(page.locator('.ant-modal-title:has-text("发布题目")')).toBeVisible();

// 在模态框内操作
const modal = page.locator('.ant-modal');
await modal.locator('input#title').fill('测试题目');
await modal.getByRole('button', { name: '提交' }).click();

// 等待模态框关闭
await expect(modal).not.toBeVisible({ timeout: 3000 });
```

## 8. 常见错误和解决方案速查表

|---------|---------|---------|---------|

| **Intercepts pointer** | `element intercepts pointer` | 其他元素遮挡 | 1. 关闭模态框<br>2. 等待动画完成<br>3. 使用 `force: true` |

## 9. 测试数据隔离与唯一性

### 9.1 问题：测试数据污染导致测试失败

多次运行测试会在数据库中累积重复数据，导致选择器匹配到错误的记录。

#### ❌ 错误示例

```typescript
// 使用固定的测试数据内容
test('创建并审核题目', async ({ page }) => {
  const questionCode = await createDraftQuestion(page, {
    type: '判断题',
    subject: '数学',
    grade: '七年级',
    content: '【R405】测试审核批准功能 - 2+2=4',  // ❌ 固定内容
    level: 'L1',
    difficulty: '简单',
    score: '5'
  });

  // 后续查找题目
  const targetRow = page.locator('.ant-table-tbody tr')
    .filter({ hasText: '【R405】测试审核批准功能' })  // ❌ 会匹配到多个历史记录
    .first();  // ❌ .first() 不保证返回最新记录

  await targetRow.locator('button:has-text("发布")').click();  // ❌ 可能找到已发布的旧记录，没有"发布"按钮
});
```

**失败场景：**
1. 第一次运行：创建题目 ID=1，内容="【R405】测试审核批准功能"，测试通过 ✅
2. 第二次运行：创建题目 ID=2，内容="【R405】测试审核批准功能"
   - 数据库现在有两条相同内容的记录（ID=1 和 ID=2）
   - `.filter({ hasText: '【R405】测试审核批准功能' }).first()` 可能匹配到 ID=1（已发布的旧记录）
   - 旧记录已经不在草稿箱，没有"发布"按钮
   - 测试失败 ❌ TimeoutError: waiting for button "发布"

**根本原因：**
- 测试数据不唯一，每次运行创建相同内容的记录
- `.first()` 选择器不保证返回最新创建的记录（可能按 DOM 顺序，不是时间顺序）
- 数据库中历史数据干扰了当前测试

#### ✅ 正确示例

```typescript
// 方案1: 使用时间戳确保数据唯一性（推荐）
test('创建并审核题目', async ({ page }) => {
  // 为每次测试运行生成唯一标识

  const uniqueContent = `【R405-${timestamp}】测试审核批准功能 - 2+2=4`;

    type: '判断题',
    subject: '数学',
    grade: '七年级',
    content: uniqueContent,  // ✅ 每次运行都不同
    level: 'L1',

    score: '5'
  });

  // 使用唯一内容查找，确保找到本次测试创建的记录
  const targetRow = page.locator('.ant-table-tbody tr')
    .filter({ hasText: `【R405-${timestamp}】` })  // ✅ 只会匹配本次创建的记录
    .first();

  await targetRow.locator('button:has-text("发布")').click();  // ✅ 必定找到草稿状态的记录
});
```

### 9.2 跨测试步骤的数据传递

当测试涉及多个用户或多个页面时，需要在整个测试中传递唯一标识。

#### ✅ 正确示例

```typescript
test('双用户审核流程', async ({ page }) => {
  // Step 1: 在测试开始时生成唯一标识
  const timestamp = Date.now();
  const uniqueContent = `【R405-${timestamp}】测试审核批准功能 - 2+2=4`;

  // Step 2: 用户1创建题目
  console.log('Step 1: teacher01 登录');
  await loginAsTeacher(page, 'teacher01', 'password');

  const questionCode = await createDraftQuestion(page, {
    content: uniqueContent,  // ✅ 使用唯一内容
    // ...
  });

  // Step 3: 用户1提交审核
  console.log(`查找题目内容: 【R405-${timestamp}】`);
  const draftRow = page.locator('.ant-table-tbody tr')
    .filter({ hasText: `【R405-${timestamp}】` })  // ✅ 使用相同的时间戳
    .first();
  await draftRow.locator('button:has-text("发布")').click();

  // Step 4: 用户2审核
  console.log('Step 2: teacher02 登录');
  await loginAsTeacher(page, 'teacher02', 'password');

  // ✅ 用户2也使用相同的时间戳查找题目
  console.log(`查找待审核题目: 【R405-${timestamp}】`);
  const reviewRow = page.locator('.ant-table-tbody tr')
    .filter({ hasText: `【R405-${timestamp}】` })  // ✅ 确保找到同一题目
    .first();
  await reviewRow.locator('button:has-text("审核")').click();
});
```

### 9.3 关键原则

**数据唯一性：**
1. **使用时间戳或 UUID** - 确保每次测试运行的数据都是唯一的
2. **在测试开始时生成** - 在 test() 函数顶部生成唯一标识
3. **在所有步骤中使用** - 创建、查找、验证都使用相同的唯一标识
4. **记录到日志** - `console.log()` 输出唯一标识，方便调试

**选择器策略：**
1. **避免使用 .first() 处理重复数据** - .first() 不保证顺序
2. **使用唯一标识符过滤** - 时间戳、UUID、或测试专用字段
3. **验证找到的记录** - 使用 `expect().toHaveCount(1)` 确保只匹配一条

**数据清理：**
1. **测试前清理（推荐）** - 使用 `beforeEach()` 清理历史数据
2. **测试后清理** - 使用 `afterEach()` 清理本次数据（可选）
3. **定期清理** - 定期清理测试数据库，避免累积

### 9.4 实际案例

**案例：R405 测试修复**

**问题描述：**
测试 R405 在第一次运行时通过，但后续运行失败，错误为"TimeoutError: waiting for button 发布"

**问题分析：**
```typescript
// 问题代码
const targetRow = page.locator('.ant-table-tbody tr')
  .filter({ hasText: '【R405】测试审核批准功能' })
  .first();  // 匹配到数据库中的多条记录，.first() 可能返回旧记录

await targetRow.locator('button:has-text("发布")').click();  // 旧记录已发布，没有"发布"按钮
```

**数据库状态：**
| ID | 内容 | 状态 | 创建时间 |
|----|------|------|---------|
| 214 | 【R405】测试审核批准功能 | 已发布 | 第1次运行 |
| 216 | 【R405】测试审核批准功能 | 已发布 | 第2次运行 |
| 217 | 【R405】测试审核批准功能 | 已发布 | 第3次运行 |
| 222 | 【R405】测试审核批准功能 | 草稿 | 第4次运行 ← 当前测试 |

`.filter({ hasText: '【R405】测试审核批准功能' }).first()` 可能匹配到 ID=214（最早的记录），而不是 ID=222（最新的草稿）。

**解决方案：**
```typescript

const timestamp = Date.now();

const questionCode = await createDraftQuestion(page, {
  content: uniqueContent,  // ✅ 唯一内容
  // ...
});

// 创建时使用时间戳
const targetRow = page.locator('.ant-table-tbody tr')
  .filter({ hasText: `【R405-${timestamp}】` })  // ✅ 只匹配本次测试
  .first();

// 审核时也使用相同时间戳
const reviewRow = page.locator('.ant-table-tbody tr')
  .filter({ hasText: `【R405-${timestamp}】` })  // ✅ 确保找到同一题目
  .first();
```

**修复后的数据库状态：**
| ID | 内容 | 状态 | 创建时间 |
|----|------|------|---------|
| 214 | 【R405】测试审核批准功能 | 已发布 | 旧数据 |
| 222 | 【R405-1761045256357】测试审核批准功能 | 草稿 | 第N次运行 ✅ |
| 223 | 【R405-1761045501771】测试审核批准功能 | 草稿 | 第N+1次运行 ✅ |

现在每次运行都创建唯一记录，`.filter()` 只会匹配当前测试的记录，测试稳定通过。

**关键要点：**
- ✅ 时间戳确保数据唯一性
- ✅ 跨步骤使用相同标识符
- ✅ 避免 .first() 匹配历史数据
- ✅ 日志输出帮助调试

## 10. 页面导航测试策略

### 10.1 问题：直接URL跳转隐藏导航问题

使用 `page.goto()` 直接跳转URL无法发现页面导航菜单的问题。

#### ❌ 错误示例

```typescript
// 直接URL跳转
test('访问题库页面', async ({ page }) => {
  await page.goto('/teacher/question-bank');  // ❌ 跳过了导航菜单
  await page.waitForLoadState('networkidle');

  // 测试通过，但没有测试导航菜单
});
```

**隐藏的问题：**
- 导航菜单可能不存在（UI bug）
- 导航链接可能指向错误的路径
- 导航菜单可能没有正确的权限控制

#### ✅ 正确示例

```typescript
// 方案1: 使用点击导航（推荐）
test('通过导航访问题库页面', async ({ page }) => {
  // 先登录到首页
  await loginAsTeacher(page, 'teacher01', 'password');
  await page.waitForLoadState('networkidle');

  // 通过点击导航菜单进入页面
  const menuLink = page.locator('a:has-text("题库管理")');
  await expect(menuLink).toBeVisible();  // ✅ 验证菜单存在
  await menuLink.click();  // ✅ 模拟真实用户操作

  // 验证URL跳转正确
  await page.waitForURL(/\/teacher\/question-bank/);  // ✅ 验证导航正确
  await page.waitForLoadState('networkidle');

  // 验证页面内容加载
  await expect(page.locator('h1:has-text("题库管理")')).toBeVisible();
});

// 方案2: 创建导航辅助函数（最佳实践）
const navigateToQuestionBank = async (page: Page) => {
  console.log('通过点击导航菜单进入题库页面');

  // 等待页面完全加载
  await page.waitForLoadState('networkidle');
  await page.waitForTimeout(500);

  // 检查当前URL，避免重复导航
  const currentUrl = page.url();
  if (currentUrl.includes('/teacher/question-bank') ||
      currentUrl.includes('/admin/question-bank')) {
    console.log('当前已在题库页面，无需重新导航');
    return;
  }

  // 尝试多种选择器找到题库管理菜单（提高鲁棒性）
  const menuSelectors = [
    'a:has-text("题库管理")',              // 链接文本
    '.ant-menu-item:has-text("题库管理")',  // Ant Design 菜单项
    'span:has-text("题库管理")',            // 菜单文本
    '[href*="question-bank"]'              // href 包含 question-bank
  ];

  let menuFound = false;
  for (const selector of menuSelectors) {
    const menu = page.locator(selector).first();
    if (await menu.count() > 0 && await menu.isVisible()) {
      console.log(`找到题库管理菜单: ${selector}`);
      await menu.click();
      menuFound = true;
      break;
    }
  }

  if (!menuFound) {
    throw new Error('无法找到题库管理菜单链接');
  }

  // 等待页面跳转完成
  await page.waitForURL(/\/(teacher|admin)\/question-bank/, { timeout: 5000 });
  await page.waitForLoadState('networkidle');

  console.log('成功导航到题库页面');
};

// 使用辅助函数
test('测试题库功能', async ({ page }) => {
  await loginAsTeacher(page, 'teacher01', 'password');
  await navigateToQuestionBank(page);  // ✅ 封装导航逻辑

  // 继续测试...
});
```

### 10.2 导航辅助函数的好处

**优势：**
1. **发现UI问题** - 能够发现导航菜单缺失或配置错误
2. **模拟真实用户** - 测试用户真实的操作路径
3. **提高覆盖率** - 测试导航功能，不仅仅是目标页面
4. **避免重复导航** - 检查当前URL，已在目标页面则跳过
5. **多选择器策略** - 提高测试的鲁棒性
6. **清晰的日志** - 便于调试和理解测试流程

**实际案例：**
在 R405/R406 测试修复中，使用点击导航发现了教师角色缺少顶部导航菜单的问题：
- 问题：教师登录后没有导航菜单，只能通过直接URL访问页面
- 发现：切换到点击导航后，测试失败 `TimeoutError: waiting for menu`
- 修复：在 `MainLayout.tsx` 中为教师角色添加导航菜单
- 结果：提升了用户体验，统一了管理员和教师的界面

### 10.3 关键原则

**导航测试：**
1. **优先使用点击导航** - 而不是 `page.goto()`
2. **验证导航元素存在** - 检查菜单链接是否可见
3. **验证URL跳转正确** - 使用 `waitForURL()` 确认路径
4. **封装导航逻辑** - 创建可复用的辅助函数
5. **处理已导航场景** - 检查当前URL，避免重复操作

**何时使用直接URL跳转：**
- ✅ 测试初始化（如登录后的首页）
- ✅ 测试前的数据准备
- ✅ 性能测试（减少不必要的导航步骤）
- ❌ 功能测试中的页面切换（应使用点击导航）

## 11. 总结

### 核心原则

1. **优先使用语义化选择器** - `getByRole`, `getByLabel`, `getByText`
2. **先检查附加状态** - 使用 `toBeAttached()` 而不总是 `toBeVisible()`
3. **给 UI 足够的时间** - 等待网络、动画、模态框
4. **从父到子定位** - 先找到容器，再找子元素
5. **处理边界情况** - 检查元素是否存在、是否可见、是否可操作
6. **使测试可维护** - 提取公共逻辑，使用配置常量

### 常见陷阱

1. ❌ 不等待页面加载就操作
2. ❌ 使用脆弱的文本选择器
3. ❌ 忽略 Ant Design 的虚拟滚动
4. ❌ 直接 `check()` Checkbox 不检查状态
5. ❌ 不处理模态框和遮挡层
6. ❌ 硬编码超时时间
7. ❌ 不处理异步操作

### 提高初次通过率的关键

1. **理解 UI 框架特性** - Ant Design 的虚拟滚动、动画、状态管理
2. **使用稳定的选择器** - Role-based > Label-based > CSS > Text
3. **合理的等待策略** - 不要太短也不要太长
4. **充分的错误处理** - 预见可能的失败场景
5. **本地验证** - 在提交前本地运行多次
