## 测试概述

本项目的测试分为**两个主要类别**：

1. **E2E测试 (End-to-End Tests)** - 基于Playwright的前端功能测试
2. **API测试 (API Tests)** - 后端接口功能测试

### 测试技术栈
- **E2E测试框架**: Playwright + TypeScript
- **API测试**: Node.js + Axios / HTTP模块
- **测试文档**: CSV格式测试用例追踪

---

## 测试级别

本项目包含三种测试级别，分别关注不同的测试目标：

### 1️⃣ 冒烟测试 (Smoke Tests)

**目标**: 快速验证系统核心功能是否正常工作

**执行时间**: 约 2-5 分钟

**执行方式**: 并行执行

**覆盖范围**:
- ✅ 学生/教师/管理员登录
- ✅ 首页基本显示
- ✅ API健康检查
- ✅ 数据库连接

**测试文件位置**:
- E2E: `tests/e2e/smoke/smoke.spec.ts`
- API: `tests/api/smoke-test.js`

**执行命令**:
```bash
# E2E冒烟测试
npx playwright test tests/e2e/smoke/ -c tests/playwright.config.ts

# API冒烟测试
node tests/api/smoke-test.js
```

---

### 2️⃣ 回归测试 (Regression Tests)

**目标**: 验证系统各功能模块，确保新代码不会破坏现有功能

**执行方式**: 可并行执行（测试用例之间独立）

**执行时间**: 约 10-20 分钟

**覆盖范围**:

#### 认证模块 (R001-R007)
- 学生/教师登录（正常、异常流程）
- 输入验证（身份证格式、空字段）
- Tab 切换功能

#### 学生功能模块 (R101-R105)
- 首页导航、考试列表
- 成绩查询、个人信息
- 证书验证

#### 管理员功能模块 (R201-R205)
- 管理后台访问
- 用户/考试/题库管理
- 成绩统计

#### 题库创建模块 (R301-R310)
- 7种题型创建（单选、多选、填空、判断、问答、编程、匹配）
- 表单验证和动态功能
- 题型切换

**测试文件位置**:
- `tests/e2e/regression/auth.spec.ts` - 认证模块
- `tests/e2e/regression/student-features.spec.ts` - 学生功能
- `tests/e2e/regression/admin-features.spec.ts` - 管理功能
- `tests/e2e/regression/question-bank-creation.spec.ts` - 题库创建

**执行命令**:
```bash
# 运行所有回归测试
npx playwright test tests/e2e/regression/ -c tests/playwright.config.ts

# 运行特定回归测试
npx playwright test tests/e2e/regression/question-bank-creation.spec.ts -c tests/playwright.config.ts
```

---

### 3️⃣ 工作流测试 (Workflow Tests)

**目标**: 验证完整的业务流程，确保端到端的用户操作路径正常运行

**执行时间**: 约 5-15 分钟

**执行方式**: **必须顺序执行**（测试用例之间有依赖关系）

**特点**:
- ✅ 测试完整的用户业务流程（从开始到结束）
- ✅ 测试用例之间有严格的依赖关系和执行顺序
- ✅ 模拟真实用户使用场景
- ✅ 验证多个功能模块的协作

**覆盖的业务流程**:

#### 学生注册流程 (REG101-REG111)
完整流程：学生注册 → 管理员审批 → 学生登录 → 使用功能

测试步骤：
1. REG101-REG104: 学生注册和状态查询
2. REG105-REG107: 管理员审批流程
3. REG108-REG109: 学生登录和功能访问
4. REG110: 负面测试（重复注册）
5. REG111: 管理员删除学生账号

#### 题库审核流程 (REV101-REV108)
完整流程：创建草稿 → 提交审核 → 审核批准/拒绝 → 发布使用

测试步骤：
1. 教师创建题目草稿
2. 提交审核申请
3. 审核人员审批
4. 发布到题库

**测试文件位置**:
- `tests/e2e/regression/student-registration.spec.ts` - 学生注册流程
- `tests/e2e/regression/question-bank-review.spec.ts` - 题库审核流程

**执行命令**:
```bash
# ⚠️ 重要：工作流测试必须使用 --workers=1 顺序执行
npx playwright test tests/e2e/regression/student-registration.spec.ts -c tests/playwright.config.ts --workers=1

# 运行特定工作流测试
npx playwright test tests/e2e/regression/student-registration.spec.ts --grep "REG10" -c tests/playwright.config.ts --workers=1
```

**注意事项**:
- ⚠️ 工作流测试 **不能并行执行**，必须使用 `--workers=1`
- ⚠️ 测试用例有严格的执行顺序，不能单独运行中间步骤
- ⚠️ 如果某个测试失败，后续依赖的测试也会失败
- ✅ 适合用于验证完整业务流程的正确性
- ✅ 适合用于端到端的集成测试

### 测试报告目录结构

所有测试结果统一存放在 `tests/test-results/` 目录：

```
tests/
├── test-results/                    # 所有测试结果统一目录
│   ├── html/                        # HTML报告
│   │   ├── index.html              # 报告入口
│   │   └── data/                   # 报告数据
│   ├── artifacts/                   # 测试artifacts（截图、视频、trace）
│   │   └── [test-name]/            # 每个测试的artifacts
│   │       ├── test-failed-1.png   # 失败截图
│   │       ├── video.webm          # 测试录像
│   │       └── trace.zip           # Playwright trace
│   └── results.json                 # JSON测试结果
```

**配置说明** (`tests/playwright.config.ts`):

```typescript
export default defineConfig({
  testDir: './e2e',

  // 测试artifacts输出目录（截图、视频、trace等）
  outputDir: '../tests/test-results/artifacts',

  // 测试报告
  reporter: [
    ['html', { outputFolder: '../tests/test-results/html' }],
    ['json', { outputFile: '../tests/test-results/results.json' }],
    ['list']
  ],

  use: {
    baseURL: 'http://localhost:80',
    trace: 'on-first-retry',      // 失败重试时记录trace
    screenshot: 'only-on-failure', // 失败时截图
    video: 'retain-on-failure',    // 失败时保留视频
  },
});
```

**路径说明**:
- 配置文件位于 `tests/playwright.config.ts`
- 测试从项目根目录运行，路径使用 `../tests/test-results/` 回到根目录后进入正确路径
- 所有输出都集中在 `tests/test-results/` 目录，便于管理和清理

**清理测试结果**:
```bash

# 下次运行测试会自动重新创建目录
```

## 测试最佳实践

### 1. 测试编写原则

✅ **DO 应该做**:
- 使用描述性的测试名称
- 每个测试只验证一个功能点
- 添加适当的等待和超时处理
- 测试应该是独立和幂等的
- 使用测试配置文件中的常量

❌ **DON'T 不应该做**:
- 测试之间有依赖关系
- 使用硬编码的等待时间（避免固定延迟）
- 忽略错误处理
- 测试数据硬编码在测试中

### 2. 选择器策略

优先级顺序:
1. **data-testid** 属性（最优）
2. **文本内容** 选择器
3. **角色和标签** 选择器
4. **CSS类名**（最后选择）

```typescript
// 好的选择器
await page.locator('[data-testid="login-button"]').click();
await page.getByRole('button', { name: '登录' }).click();
await page.getByText('提交').click();

// 避免的选择器
await page.locator('.btn-primary').click(); // 易变
```

### 3. 等待策略

```typescript
// 等待网络空闲
await page.waitForLoadState('networkidle');

// 等待元素可见
await expect(page.locator('.element')).toBeVisible({ timeout: 5000 });

// 等待API响应
await page.waitForResponse(response =>
  response.url().includes('/api/auth/login') && response.status() === 200
);
```

### 4. 错误处理

```typescript
// 检查元素是否存在
const elementCount = await page.locator('.element').count();
if (elementCount > 0) {
  await page.locator('.element').first().click();
}

// Try-catch处理
try {
  await page.click('.optional-button', { timeout: 2000 });
} catch (error) {
  console.log('Button not found, skipping');
}
```

### 5. 测试维护

- **定期更新**: 功能变更时同步更新测试
- **失败处理**: 测试失败时及时修复
- **代码审查**: 测试代码也需要 Code Review
- **文档更新**: 更新测试用例文档(CSV)

## 故障排除

### 常见问题

#### 1. Playwright浏览器下载失败

```bash
# 使用国内镜像
PLAYWRIGHT_DOWNLOAD_HOST=https://playwright.download.prss.microsoft.com npx playwright install
```

#### 2. 测试超时

**原因**: 服务未启动或响应慢

**解决方案**:
```bash
# 检查Docker服务状态
docker-compose ps

# 查看服务日志
docker-compose logs backend
docker-compose logs frontend

# 重启服务
docker-compose restart
```

#### 3. 元素找不到

**原因**: 选择器不正确或页面未加载完成

**解决方案**:
```bash
# 使用Playwright Inspector检查元素
npx playwright test --debug tests/e2e/your-test.spec.ts

# 使用UI模式逐步执行
npx playwright test --ui
```

#### 4. API连接失败

**原因**: 后端服务未启动或端口冲突

**解决方案**:
```bash
# 检查后端健康
curl http://localhost:3001/health

# 检查端口占用
netstat -ano | findstr :3001  # Windows
lsof -i :3001                  # Mac/Linux

# 查看后端日志
docker-compose logs backend
```

#### 5. 数据库连接失败

```bash
# 检查PostgreSQL容器
docker-compose ps postgres

# 测试数据库连接
docker exec -it guiyang_oj_postgres psql -U postgres -d guiyang_oj -c "SELECT 1"

# 查看数据库日志
docker-compose logs postgres
```

### 测试报告目录常见问题

#### Q1: 为什么测试结果在多个目录？
**A**: 需要在 `playwright.config.ts` 中配置 `outputDir`:
```typescript
outputDir: '../tests/test-results/artifacts'
```

#### Q2: 如何确保测试结果在正确位置？
**A**: 验证步骤:
```bash
# 1. 清理旧目录
rm -rf test-results tests/tests

# 2. 运行一个测试
npx playwright test -c tests/playwright.config.ts --grep "R401"

# 3. 检查目录 - 应该只看到一个
find . -type d -name "test-results"
# 输出应该是: ./tests/test-results
```

#### Q3: 相对路径如何配置？
**A**: Playwright配置中的路径相对于**当前工作目录**（项目根目录），不是配置文件:
```typescript
// 配置文件在: tests/playwright.config.ts
// 工作目录在: D:\CS\Git (项目根目录)
// 因此使用: ../tests/test-results/ 来回到根目录再进入正确路径

export default defineConfig({
  outputDir: '../tests/test-results/artifacts',  // 正确 ✅
  // 不是: 'tests/test-results/artifacts'         // 错误 ❌（会变成 tests/tests/...）
  // 不是: './test-results/artifacts'             // 错误 ❌（会在根目录）
});
```

#### Q4: 始终从哪里运行测试？
**A**: 始终从项目根目录运行:
```bash
# ✅ 正确 - 从项目根目录
cd /d/CS/Git
npx playwright test tests/e2e/... -c tests/playwright.config.ts

# ❌ 错误 - 从tests目录运行会导致路径混乱
cd /d/CS/Git/tests
npx playwright test e2e/... -c playwright.config.ts
```

## 附录：测试报告目录配置详解

### Playwright路径配置原理

Playwright配置文件中的路径遵循以下规则：

1. **testDir**: 相对于配置文件位置
   ```typescript
   testDir: './e2e'  // 相对于 tests/playwright.config.ts -> tests/e2e/
   ```

2. **outputDir 和 reporter paths**: 相对于**当前工作目录**(CWD)
   ```typescript
   // 工作目录: D:\CS\Git (项目根目录)
   outputDir: '../tests/test-results/artifacts'
   // 解析: tests/playwright.config.ts位置 -> 向上一级 -> tests/test-results/artifacts
   ```

3. **运行命令**: 始终从项目根目录执行
   ```bash
   pwd  # D:\CS\Git
   npx playwright test tests/e2e/... -c tests/playwright.config.ts
   ```

### 完整配置示例

**文件**: `tests/playwright.config.ts`

```typescript
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './e2e',  // 相对于配置文件

  // 测试artifacts输出目录（截图、视频、trace）
  outputDir: '../tests/test-results/artifacts',  // 相对于工作目录

  // 测试报告配置
  reporter: [
    ['html', { outputFolder: '../tests/test-results/html' }],
    ['json', { outputFile: '../tests/test-results/results.json' }],
    ['list']
  ],

  use: {
    baseURL: 'http://localhost:80',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },
});
```

### 路径解析示例

**错误示例 1**: 使用 `tests/test-results/`
```
工作目录: D:\CS\Git
配置路径: tests/test-results/html
实际路径: D:\CS\Git\tests\test-results\html  ❌
问题: 从根目录看已经进入tests了，再加上tests会重复
```

**错误示例 2**: 使用 `./test-results/`
```
工作目录: D:\CS\Git
配置路径: ./test-results/html
实际路径: D:\CS\Git\test-results\html  ❌
问题: 结果在根目录，不在tests目录下
```

**正确示例**: 使用 `../tests/test-results/`
```
配置文件: tests/playwright.config.ts
向上一级: ../ -> D:\CS\Git
进入目录: tests/test-results/html
实际路径: D:\CS\Git\tests\test-results\html  ✅
```

### 验证配置正确性

```bash
# 步骤1: 清理所有旧的测试结果目录
rm -rf test-results tests/tests/test-results

# 步骤2: 运行一个简单测试
npx playwright test -c tests/playwright.config.ts --grep "R401"

# 步骤3: 检查目录结构
find . -type d -name "test-results"
# 期望输出: ./tests/test-results （只有一个）

# 步骤4: 验证文件存在
ls tests/test-results/
# 期望看到: artifacts/  html/  results.json

# 步骤5: 检查artifacts
ls tests/test-results/artifacts/
# 应该看到测试相关的目录
```

### 使用绝对路径（备选方案）

如果相对路径难以理解，可以使用绝对路径：

```typescript
import path from 'path';
import { defineConfig } from '@playwright/test';

export default defineConfig({
  outputDir: path.join(__dirname, '../test-results/artifacts'),
  reporter: [
    ['html', {
      outputFolder: path.join(__dirname, '../test-results/html')
    }],
    ['json', {
      outputFile: path.join(__dirname, '../test-results/results.json')
    }],
  ],
});
```
